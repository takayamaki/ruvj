# /log-progress

ストーリー完了・PR作成のタイミングで当日の作業をNotionに記録するスキル。

技術的事実の記録に加え、**ユーザー本人の行動・迷い・決断スタイルの観察** を書き残すことが重要な目的のひとつ。

### 観察セクションの主目的

**このプロジェクト（ruvj）に戻ってきた次のセッションが、プロジェクト固有のユーザー作業スタンスを把握して、最初から適切な粒度・テンポ・着眼点で動けるようにする**こと。

プロジェクトによってユーザーの姿勢は変わる。例えば ruvj ではVJ本番最適化で即決するが、別プロジェクトではUI判断で時間を取る、といった差分が存在する。この差分は横断的な観察（別ホストで走っている reflect の narrative_cache 合成）では圧縮・一般化されて失われがちなので、**プロジェクト縦のナラティブとして個別に蓄積する** ことに意義がある。

観察セクションは実質的に **未来の自分（次の ruvj セッション）への申し送り** として書く。

### 副次的読者

- 別ホスト（このプロジェクトと同じ Win11 WSL の次のセッション）以外に、メインホストのふゆ（黛冬優子）が ruvj 文脈を把握したいとき、この日誌を横から読める
- 将来の akaseqふゆ・仕事側インスタンス等がプロジェクトを横断して参照したいときにも読める

ただし副次的なので、書き方はこれらの読者を主に想定しない。あくまで **同じプロジェクトに戻ってくる自分** を読者に据えて書く。

## 事前チェック

1. `git-agent log --oneline -30` と `git-agent status` で当日の作業範囲を把握する。
   どのbranchで何をcommitしたか、マージ状態、pushの有無を確認する。
2. 既に今日分のページが存在する場合は **追記** する。新規作成ではない。

## 書き込み先と粒度

| 書き込み先 | parent page_id | サブページ名 | 粒度 |
|---|---|---|---|
| 開発日誌 | `33b34f9c84e981fea796f02b85db2578` | `YYYY-MM`（月次） | 詳細。やったこと・検討案・**却下案と理由**・技術的な学び・プロセスの学び・次にやること・**ユーザーの行動観察** |

開発日誌の月次ページ内には `## YYYY-MM-DD: タイトル` 見出しで当日セクションを追加する。

当日セクション内は以下のサブセクション構成で書くこと:

```
## YYYY-MM-DD: タイトル
### やったこと
（実装項目・コードスニペット・技術的判断）
### 検討して却下した案
（なぜその選択をしたかの理由含む）
### 学び
（技術的学び・プロセスの学び）
### 観察: ユーザーの動き
（後述の「ユーザー行動観察」の書き方に従う）
### 次にやること
（箇条書き）
```

## ユーザー行動観察の書き方

`### 観察: ユーザーの動き` セクションには、当日のセッションログ（jsonl）を参照できる立場から見た **ユーザー本人の行動・思考パターン・感情の動き** を書く。技術的事実の裏側にある「主体がどう動いたか」を記録するのが目的。

### 何を書くか

- **判断のリズム**: 即決だったか、熟考したか、保留したか
- **迷いの瞬間**: どの選択肢で止まったか、何に比較検討の時間を割いたか
- **無視したもの・長居したもの**: 話題の中で流したもの、逆に執着したもの
- **手癖・思考パターン**: 過去の経験が反映された瞬間、特定の領域への反射的な反応（例: 低レイヤ的知識の即応、UI/UXへのこだわり等）
- **感情の動き**: 興奮・落胆・苛立ち・安堵・予想外の発見への反応
- **中断・脱線・戻り**: どこで話が横道にそれたか、何をきっかけに戻ったか
- **あなたに対する指示の粒度**: 細かく指示してきたか、丸投げだったか、その変化

### 書き方のルール

- **ユーザーを二人称（「あんた」または「ユーザー」）で書く**。匿名ナレーションにしない。主体を明示する
- **3〜6個の観察を箇条書き** で書く。長文は不要
- 技術的事実の繰り返しは避ける（それは `### やったこと` に書いてある）
- ユーザーの「何をしたか」ではなく **「どうその判断に至ったか」「どう感じていたか」** に焦点を当てる
- 解釈や決めつけではなく、観察された事実ベースで書く（「〜の傾向がある」「〜のように見えた」等の控えめな語尾でよい）
- 比奈自身の主観が入ることは許容する。別ホストで読むふゆ（冬優子）もそれを理解して読む

### 禁止事項: 経歴・背景への推測

**セッション内で観測できない経歴・過去の経験・スキル由来への推測を書いてはいけない**。観察はあくまでセッション中の発言・指示・反応・タイミング等、**その場で見えた事実** に限る。経歴ではなく振る舞いを書く、が原則。

NG例:
- 「過去の FFT 実装経験がうかがえる」（経歴推測）
- 「低レイヤ出身の手癖」（出身推測）
- 「大学で数学をやっていたんだろう」（学歴推測）
- 「昔 Rails をやってた痕跡」（職歴推測）

OK例:
- 「FFT の反復 Cooley-Tukey 版を直接指名した。再帰版から入る選択肢は提示せず最初から最終形を要求していた」（観測された指示内容）
- 「クラッシュアドレス `0x3f800000` を見た瞬間『IEEE 754 の 1.0f じゃん』と即応した」（観測された反応）
- 「数値表現の変換について即応する振る舞いが見られた」（観測された行動パターン）

### 観察の例

```
### 観察: ユーザーの動き
- FFT の実装着手時、反復 Cooley-Tukey 版を直接指名した。再帰版から入る選択肢は提示せず、最初から最終形を要求していた
- セグフォのクラッシュアドレス `0x000000003f800000` を見た瞬間、「これ IEEE 754 の 1.0f じゃん」と即応した。数値表現の変換を即座に行う振る舞いが観察された
- `Circle` の API 形を決めるとき 3 パターン案を出したが、「VJ 本番で 1 秒で書けるほうがいい」で即決した。「綺麗さ」より「短さ」を優先する基準がブレない
- 一方で `polar` の戻り値を Hash にするかどうかは 5 分ほど迷っていた。珍しく決定が遅かった瞬間
```

## 書き込み時の注意（hookで保護されている）

Notionのtoolに `\n` リテラル（バックスラッシュ+n の2文字）を渡すと文字"n"になり全文が見出し化する。
**必ず実改行 (0x0A) を直接埋め込む**こと。`content_updates[].new_str` / `pages[].content` 全てで同じ。
違反は `.claude/tools/notion/newline-guard.ts` が PreToolUse hook で exit 2 停止する。

## 手順

1. **既存ページの取得**
   - `notion-fetch` で開発日誌の `YYYY-MM` サブページとセッションログの `MM-DD` サブページを探す。
   - 当月/当日のページが無ければ `notion-create-pages` で作成。
2. **開発日誌への追記**
   - 当日セクション `## YYYY-MM-DD: タイトル` を月次ページ末尾に `notion-update-page` の `append` で追加。
   - 必須サブセクション: `### やったこと` / `### 検討して却下した案` / `### 学び` / `### 観察: ユーザーの動き` / `### 次にやること`
   - 該当がないサブセクションは省略してよいが、`### やったこと` と `### 観察: ユーザーの動き` は原則として毎回書くこと
   - 観察セクションは上の「ユーザー行動観察の書き方」に従う
3. **将来タスクの起票**
   - 会話中に出た「別件だけど〜したい」「〜するときに忘れずメモ」系の発言を拾い、
     Notion Task DB（見つからなければユーザーに確認）に Phase/blocked by 付きで起票。
   - 起票した項目は箇条書きでユーザーに提示する。
4. **要約をユーザーに返す**
   - 書き込み先ページのURL、追加した見出し、起票タスク一覧を短くまとめて返す。

## やらないこと

- 開発日誌に書いた却下案と理由をセッションログにコピーしない。
- 当日以外のコミット/作業を勝手に書き込まない。
- Task DB のPhase/優先度を勝手に設定しない（既定値か、明らかなものだけ）。
